# /project:loop-task

适用于需要精确实现且支持自动验证的功能开发或 Bug 修复。调用后按 4 步流程推进，第 4 步失败则自动进入下一轮修正。

---

## 使用方式

在任务描述前加上 `/project:loop-task`，例如：

```
/project:loop-task 给宠物详情页添加体重编辑功能
```

## [流程]

### 1. 分析现有代码

- 复杂功能使用 `plan-dev` agent 做方案设计，输出实施计划
- 简单修改自行定位涉及文件
- 理清数据流：输入 → 处理 → 输出 的完整链路
- 标注关键函数/类型/接口的签名和现有行为
- 输出分析结论：问题根因或实现切入点

### 2. 编写测试用例

- 在正式的测试文件中编写（Go 用 `_test.go`）
- 覆盖：
  - 正常路径（happy path）
  - 边界条件（空值、极值、特殊字符）
  - 错误路径（非法输入、依赖失败、权限不足）
- 测试用例与第 1 步的分析结论对应：每个根因/切入点至少一个测试

### 3. 实施修改

- 严格遵循项目的编码规则（参见 `rules/` 目录）
- 修改后执行：
  - 后端：`cd backend && go vet ./...`
  - 前端：`cd miniprogram && npx vue-tsc --noEmit`
- 确保通过编译/类型检查

### 4. 验证测试结果

- 后端运行 `cd backend && go test ./...`
- **如果测试全部通过** → 进入第 5 步代码审查
- **如果存在失败** → 进入纠错机制

### 5. 代码审查

- 使用 `code-reviewer` agent 审查变更
- 审查维度：正确性、安全性、代码质量、前后端联动、数据库同步
- **审查通过** → 完成，输出修改摘要
- **审查发现问题** → 进入纠错机制

---

## [纠错机制]

⚠️ **第 4 步（测试）或第 5 步（审查）失败时严格遵循以下流程：**

| # | 动作 | 说明 |
|---|---|---|
| ① | 读取报错 | 完整复制错误输出，不截断不概括 |
| ② | 分析根因 | 只基于报错信息，区分：逻辑错误 / 测试用例错误 / 遗漏边界 |
| ③ | 禁止重复 | 上次改 A 导致的失败，这轮不走 A 路径 |
| ④ | 回退修正 | 测试失败 → 回到对应步骤；审查发现问题 → 回第 3 步修复 |
| ⑤ | 标记迭代 | 完成后输出 `🔄 第 N 轮修正` |

### 核心约束

| 规则 | 说明 |
|---|---|
| 最大迭代 | 不超过 5 轮 |
| 禁止猜测 | 报错信息不足时，补充日志后重新运行，不得凭空推断 |
| 禁止死循环 | 同一错误路径连续两次 → 终止并报告无法自动修复 |
| 逐轮收敛 | 每轮只改一个维度的问题，不试图一次性修复所有 |
