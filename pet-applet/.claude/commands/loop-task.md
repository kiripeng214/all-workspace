# /project:loop-task

全流程：**提出需求 → 输出 PRD → 输出 Plan → 进入实现**。

适用于从需求到交付的完整链路。支持两种实现模式：

| 模式 | 适用场景 | 耗时 |
|------|---------|------|
| **顺序模式**（默认） | 单领域任务，仅后端或仅前端 | 串行执行，5 步走完 |
| **并行模式** | 跨领域任务，同时涉及后端 + 前端 + 数据库 | 子任务并行，整体更快 |

> 判断标准：如果任务涉及 **2 个及以上领域**（如后端 + 前端），使用并行模式。

## 使用方式

在任务描述前加上 `/project:loop-task`，例如：

```
/project:loop-task 给宠物详情页添加体重编辑功能
```

## [前置阶段 - 文档驱动]

任何需求在执行前先走文档流程，确保方向和方案达成一致。

### 阶段 1: 提出需求

明确需求描述，包含：
- 要做什么功能
- 解决什么痛点
- 期望效果

### 阶段 2: 输出 PRD

按照 `doc/prd/TEMPLATE.md` 编写产品需求文档，存入 `doc/prd/`：

- 背景与动机
- 功能列表与优先级（P0/P1）
- 用户流程
- 非功能性需求
- 验收标准

**PRD 完成标准**：功能列表明确，验收标准可衡量。

### 阶段 3: 输出 Plan

按照 `doc/plan/TEMPLATE.md` 编写技术方案，存入 `doc/plan/`：

- 方案概述（涉及领域）
- 涉及文件清单
- 实施步骤（含依赖关系标注）
- 风险与回退策略
- 关联 PRD 文件路径

**Plan 完成标准**：文件清单完整，实施步骤可执行，风险已识别。

> PRD 和 Plan 写入后，进入下面的实现流程。

## [流程]

### 1. 分析现有代码

- 复杂功能使用 `plan-dev` agent 做方案设计，输出实施计划
- 简单修改自行定位涉及文件
- 理清数据流：输入 → 处理 → 输出 的完整链路
- 标注关键函数/类型/接口的签名和现有行为
- 输出分析结论：问题根因或实现切入点

### 2. 编写测试用例

- 在正式的测试文件中编写（Go 用 `_test.go`）
- 覆盖：
  - 正常路径（happy path）
  - 边界条件（空值、极值、特殊字符）
  - 错误路径（非法输入、依赖失败、权限不足）
- 测试用例与第 1 步的分析结论对应：每个根因/切入点至少一个测试

### 3. 实施修改

- 严格遵循项目的编码规则（参见 `rules/` 目录）
- 修改后执行：
  - 后端：`cd backend && go vet ./...`
  - 前端：`cd miniprogram && npx vue-tsc --noEmit`
- 确保通过编译/类型检查

### 4. 验证测试结果

- 后端运行 `cd backend && go test ./...`
- **如果测试全部通过** → 进入第 5 步代码审查
- **如果存在失败** → 进入纠错机制

### 5. 代码审查

- 使用 `code-reviewer` agent 审查变更
- 审查维度：正确性、安全性、代码质量、前后端联动、数据库同步
- **审查通过** → 完成，输出修改摘要
- **审查发现问题** → 进入纠错机制

## [流程 - 并行模式]

适用于跨领域、多 agent 并行执行的任务。使用 worktree 隔离 + 自动合并。

### 阶段 1: 方案切分 (plan-dev)

输出内容：
- **子任务清单** — 按数据库/后端/前端拆解，标注覆盖文件和依赖关系
- **API 契约** — 请求/响应类型定义（Go struct + TypeScript interface）
- **数据库 schema** — 新表或字段变更的 DDL
- **路由注册** — 新增路由路径

拆分原则：每个子任务覆盖不重叠的文件集合，后端子任务不碰前端文件。

### 阶段 2: 环境准备 (merge-manager)

- 检查是否有未提交的变更（有则 stash）
- 创建集成分支 `feat/integration/{feature}`（基于 main 最新 HEAD）
- 为每个子任务创建分支和 worktree：

```bash
git branch feat/{feature}/{subtask} main
git worktree add ../pet-applet-worktrees/{subtask} feat/{feature}/{subtask}
```

### 阶段 3: 并行执行

按依赖关系分组：

```
并行组 A（无依赖）:
  - [db-dev] → 数据库 schema + 模型
  - [uni-app-dev] → 前端页面 + API 客户端

并行组 B（依赖 db-dev 的产出）:
  - [go-dev] → 后端 handler + 路由
```

每个 agent 进入对应 worktree 独立执行，执行完提交到各自分支。

### 阶段 4: 集成合并 (merge-manager)

- 切换回主仓库，进入集成分支
- 按依赖顺序合并各子任务分支
- 处理冲突（定位到对应子任务修复后重试）
- 运行全量测试：`go vet ./...` + `go test ./...` + `vue-tsc --noEmit`
- 全量测试通过后并入 `main`（`--no-ff`）

### 阶段 5: 代码审查 (code-reviewer)

- 审查集成后的总 diff
- 审查通过 → 完成（输出修改摘要）
- 审查发现问题 → 回到阶段 3 对应子任务修复

### 阶段 6: 清理 (merge-manager)

- 删除各子任务 worktree 和临时分支
- `git worktree prune`

## [纠错机制]

⚠️ **测试或审查失败时严格遵循以下流程：**

### 模式映射

| 失败阶段 | 顺序模式对应步骤 | 并行模式对应阶段 |
|---------|----------------|-----------------|
| 测试失败 | 步骤 4: 验证测试结果 | Phase 4: 集成合并（含全量测试） |
| 审查失败 | 步骤 5: 代码审查 | Phase 5: 代码审查 |

### 纠错流程

| # | 动作 | 说明 |
|---|---|---|
| ① | 读取报错 | 完整复制错误输出，不截断不概括 |
| ② | 分析根因 | 只基于报错信息，区分：逻辑错误 / 测试用例错误 / 遗漏边界 |
| ③ | 禁止重复 | 上次改 A 导致的失败，这轮不走 A 路径 |
| ④ | 回退修正 | 测试失败 → 回到对应步骤/阶段；审查发现问题 → 回实施步骤修复 |
| ⑤ | 标记迭代 | 完成后输出 `🔄 第 N 轮修正` |

### 核心约束

| 规则 | 说明 |
|---|---|
| 最大迭代 | 不超过 5 轮 |
| 禁止猜测 | 报错信息不足时，补充日志后重新运行，不得凭空推断 |
| 禁止死循环 | 同一错误路径连续两次 → 终止并报告无法自动修复 |
| 逐轮收敛 | 每轮只改一个维度的问题，不试图一次性修复所有 |
